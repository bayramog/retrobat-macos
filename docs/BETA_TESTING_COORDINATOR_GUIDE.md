## Tester Recruitment & Selection

### Application Review Process

#### Evaluation Criteria
**Hardware Diversity** (25%):
- Different Mac models (M1, M2, M3, M4)
- Various form factors (Air, Pro, Mini, iMac)
- Different RAM configurations
- macOS version diversity

**Experience Level** (20%):
- Mix of beginners and experts
- Varied technical backgrounds
- Different gaming experience levels

**Controller Availability** (15%):
- Variety of controller brands
- Different connection methods

**Commitment & Availability** (25%):
- Time availability
- Testing period availability
- Past beta testing experience
- Quality of application

**Communication Skills** (15%):
- Clear application responses
- Detailed explanations
- Discord/email presence

#### Selection Process
1. **Initial Review**: Screen applications for basic requirements
2. **Hardware Mapping**: Create hardware diversity matrix
3. **Scoring**: Rate each application on criteria above
4. **Selection**: Choose 15-20 testers for diversity
5. **Waitlist**: Keep 5-10 backup testers

### Tester Groups

**Primary Group** (10-12 testers):
- Core testers with high availability
- Diverse hardware representation
- Strong communication skills

**Secondary Group** (5-8 testers):
- Additional hardware coverage
- Specific expertise areas
- Backup for primary group

**Reserve List** (5-10 testers):
- Replacements if needed
- Second beta round candidates

## Issue Management

### Issue Triage Process

#### Severity Definitions
**Critical** (Fix immediately):
- Prevents installation
- Blocks core functionality
- Data loss potential
- Security vulnerabilities

**High** (Fix within 3-5 days):
- Major feature broken
- No workaround exists
- Affects many users
- Significant usability impact

**Medium** (Fix before release):
- Feature partially broken
- Workaround exists
- Affects some users
- Moderate usability impact

**Low** (Fix if time permits):
- Minor issues
- Cosmetic problems
- Edge cases
- Minimal impact

#### Daily Triage
1. **Review new issues** (submitted in last 24 hours)
2. **Verify and reproduce** issues when possible
3. **Categorize** by severity and type
4. **Assign** to dev team member (if critical/high)
5. **Update** issue tracker status
6. **Respond** to reporter with confirmation

#### Issue Tracking Spreadsheet

### Issue Communication

#### To Testers
- Acknowledge receipt within 24 hours
- Provide status updates on critical/high issues
- Notify when issues are fixed
- Request re-testing when needed

#### To Dev Team
- Daily summary of new issues
- Escalate critical issues immediately
- Weekly comprehensive report
- Context and user impact for each issue

## Emergency Procedures

### Critical Bug Discovery
1. Acknowledge immediately
2. Notify dev team urgently
3. Consider pausing beta if needed
4. Communicate with all testers
5. Coordinate rapid fix and release

### Tester Issues
- Unresponsive tester: Check in after 1 week
- Problematic behavior: Address privately
- Dropout: Thank them, activate backup tester

### Build Issues
- Critical bug in build: Pull immediately
- Instruct testers to stop testing
- Issue emergency patch
- Communicate clearly about issue
